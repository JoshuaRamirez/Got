# 2026-05-18

## 00:51 UTC — composition: Lenient/Strict strictness bridge landed

Executed the first step of the plan documented in
`docs/devlog/2026-05-17.md`. Added the Lenient/Strict bridge to
`composition.DefaultEngine` and the in-graph audits Strict mode allows
without an API change.

### What changed

- **`composition.Strictness`** type with `Lenient` (zero-value,
  historical behavior) and `Strict` constants.
- **`composition.NewEngine`** stays Lenient. **`NewEngineStrict`**
  added for Strict. Both return `*DefaultEngine`; the Engine interface
  is unchanged.
- **`(*DefaultEngine).Strictness()`** accessor for callers that need
  to introspect.
- **`strictAudit`** runs in `Merge` before the governance gate when
  the engine is Strict. Audit failures short-circuit: the policy
  check is skipped. Two audits implemented today:
  - **Structural** — distinct edges in g sharing the same `(from, to)`
    pair with different edge types, when both endpoints are in the
    merged frontier.
  - **Temporal** — frontier vertices with malformed `TimeTriple`
    (`ValidTo > 0 && ValidTo < ValidFrom`).
- **`auditConflict`** type — a non-policy `Conflict` carrying a
  free-text `Detail()` string. Distinct from `policyConflict` so
  callers can route on the source via type assertion.

### What this does NOT cover

Per the decision-juncture analysis in
`docs/devlog/2026-05-17.md`, real per-side content divergence —
`Textual` (Attrs disagreement), `Trust` (TrustAnnotation deltas), or
`Schema` (different `VertexType` for the same ID) — requires the API
extension that is still open:

- Option B from the juncture: `Frontier` carries `VertexEdits` /
  `EdgeEdits`, OR
- Option C: `Merge` takes a common-ancestor graph.

Strict mode here is therefore a strengthening of the gate, not a
realization of three-way merge. It catches issues that today's Lenient
mode silently accepts when they happen to be visible in the host
graph; it cannot catch issues that exist only between left and right
because the API does not carry per-side data.

The dropped `Schema` audit was a candidate but ruled out: every edge
in `g` is already schema-validated at `WithEdge` time, so any
inadmissible edge would be a write-path bug, not a merge concern.
Adding it would either duplicate `graph.Validate` or require a custom-
schema fixture to be meaningful — that depends on extending
`graph.Graph` to expose its schema, which is a separate change.

  still fires on Unsat.

  gate; no Policy conflict added.

All Lenient tests unchanged. `go build`, `go vet`, `gofmt -l .` clean.
`go test -race ./...` green across all 17 packages.

### What's still in the plan

From `docs/devlog/2026-05-17.md`, the bridge has four steps:

1. ~~Per-medium audit hook in Merge, behind a strictness flag.~~ Done.
2. **Per-side data carrier** — Frontier with `VertexEdits` /
   `EdgeEdits`, or an ancestor-taking `MergeThreeWay`. Open.
3. **Typed resolvers** that can act per-medium (e.g.
   `TextualResolver(key, choose)`). Open.
4. **Flip the default** to Strict once production callers are ready.
   Open.

Step 1 was the smallest useful slice that doesn't change the public
interface. Steps 2–4 each change observable behavior and warrant their
own decisions.
