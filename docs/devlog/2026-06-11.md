# 2026-06-11

## MergeThreeWay facade passthrough + CLI breadth

Knocked out the two cheap, no-semantic-decision backlog items together:
exposed three-way merge through the `repo.Service` facade and widened
`cmd/got` to exercise the rest of that facade. No engine behavior
changed — both are delivery/wiring work over existing operations.

### Facade passthrough (UC-U18)

`composition.DefaultEngine.MergeThreeWay` was reachable only at the
composition layer; the facade exposed two-way `Merge` but not the
ancestor-relative variant. Added `repo.Service.MergeThreeWay`
(`internal/repo/repo.go`, `internal/repo/service.go`):

- It type-asserts the configured `composition.Engine` to
  `composition.ThreeWayMerger` (the optional capability kept off the
  core Engine interface). Engines that implement only the two-way
  contract yield the new sentinel `repo.ErrThreeWayUnsupported`.
- Like `Merge`, it leaves the host graph unchanged and returns the
  outcome in the `MergeResult` — the returned `State` is the input.

`composition.NewEngine`/`NewEngineStrict` both return `*DefaultEngine`,
which satisfies `ThreeWayMerger`, so the standard wiring (and the CLI)
get three-way for free.

### CLI breadth (UC-U19)

Added four subcommands so `cmd/got` drives the rest of the facade, not
just ingest/bind/trace:

- `revise <artifact> <new-revision>` — builds a literal DPO rule
  (L=K={anchor}, R adds a `Revision` vertex + a `derived_from` edge to
  the anchor) and applies it through `repo.Service.Revise`. The CLI
  needed a local `graph.Subgraph`/`Rule`/`Match` (in `helpers.go`)
  because `Graph.Induce` can't carry R-side elements that don't exist
  in the host yet. Anchor must be an `Artifact` — the only admissible
  `derived_from` source for a new `Revision`.
- `merge --left <v,...> --right <v,...> [--ancestor <v,...>]` — two-way
  `repo.Service.Merge` by default; with `--ancestor`, three-way
  `repo.Service.MergeThreeWay`. Reports the merged vertex set or the
  typed conflicts (kind + boundary names). Read-only: no persistence.
- `materialize <v,...> [--target manifest|manifest.json]` — projects
  the induced subgraph (`projection.InduceSpec`) and materializes it via
  `repo.Service.Materialize`, printing the bundle's paths.
- `release <v,...>` — gates a frontier through `repo.Service.Release`;
  with no policy set the governance gate is vacuously `Sat`.

### Design notes

- `merge` is the only command that exercises the new passthrough
  end-to-end (via `--ancestor`), which is a nice way to keep the facade
  method on a real user path rather than test-only.
- Kept `revise` deliberately narrow (one concrete rewrite shape) to
  avoid inventing a general DPO-rule CLI surface — that would be a
  design decision, and the point here was the cheap, decision-free PR.
