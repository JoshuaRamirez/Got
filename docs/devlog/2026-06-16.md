## 18:45 UTC — Backlog sweep PR A: repo.MergeThreeWay facade + CLI breadth

Starting a six-PR sweep of the remaining backlog. This first PR is the
cheap, high-leverage one: expose three-way merge through the facade and
broaden the CLI.

### repo.Service.MergeThreeWay

Added `MergeThreeWay(ctx, state, ancestor, left, right, ps)` to the
`repo.Service` interface and `DefaultService`. It type-asserts the wired
`composition.Engine` to `composition.ThreeWayMerger` (the default engine
satisfies it) and delegates; a non-conforming engine yields the new
`repo.ErrThreeWayUnsupported`. This makes UC-U18 reachable end-to-end
like two-way `Merge`, not just at the composition layer. No mocks
implement `repo.Service`, so adding an interface method is safe.

Tests: `TestMergeThreeWayFacadeOneSidedChange` (left change wins, no
conflict), `TestMergeThreeWayFacadeConflict` (modify/modify surfaces one
Textual conflict, no frontier), `TestErrThreeWayUnsupportedSentinel`.

### CLI breadth

Added three subcommands to `cmd/got`:

- `merge <listA> <listB>` — comma-separated vertex names → two frontiers
  → `repo.Merge`; prints merged vertices + witness or typed conflicts.
- `merge3 <ancestor> <left> <right>` — → `repo.MergeThreeWay`. Frontiers
  are ID-only `EditedFrontier`s, so content comes from the host graph
  (presence-only three-way); a one-sided deletion is honored.
- `materialize` — `InduceSpec` over all vertices → `ManifestTarget` via
  `repo.Materialize`; prints the bundle paths.

`revise` and `evaluate` were deliberately skipped: a DPO rule/match and a
domain evaluator are not naturally expressible on a flag-based CLI, and
faking them would add surface without demonstrating anything real.

Tests: `TestMergeCommand`, `TestMergeUnknownVertex`,
`TestMerge3HonorsDeletion`, `TestMaterializeCommand`.

### Docs

UC-U18 scope/related note the facade passthrough; UC-U19 main scenario
gains steps 8-10 (merge/merge3/materialize) and widened channel list.
Ledger rows UC-U18, UC-U19 refreshed to 2026-06-16.

## 19:20 UTC — Backlog sweep PR B: Strict-on-Release (UC-S21)

Closes the seam documented in `TestIntegrationTemporalConflictSurfaceArea`:
`repo.Service.Release` gates on policy only and does not run the Strict
composition audit, so a malformed `TimeTriple` (or an incompatible edge
collision) surfaces through Merge but not through Release.

### Approach: expose the audit, add an opt-in strict release

Rather than change `Release`'s semantics (UC-U07/UC-S13 intent is
policy-gating), this adds a parallel strict path and leaves the default
untouched.

- `composition.Auditor` optional interface + `*DefaultEngine.Audit(ctx,
  g, f) ([]Conflict, error)` (new `internal/composition/audit.go`). It
  runs the same in-graph `strictAudit` (structural + temporal) that
  Strict Merge runs, but standalone and *independent of the strictness
  flag* — it is an explicit always-on check. This is UC-S21.
- `repo.Service.ReleaseStrict(ctx, state, f, ps)` type-asserts the wired
  composition engine to `composition.Auditor`, runs `Audit`, and refuses
  with `repo.ErrReleaseAudit` (naming the conflict kinds) on any finding;
  otherwise it delegates to the existing `Release` gate. Returns
  `repo.ErrAuditUnsupported` if the engine is not an Auditor. Plain
  `Release` is unchanged.

### Tests

- composition: `TestAuditorCapability`, `TestAuditDetectsTemporal`,
  `TestAuditClean`.
- repo: `TestReleaseStrictBlocksOnTemporal` (plain Release accepts,
  ReleaseStrict refuses), `TestReleaseStrictAllowsClean`,
  `TestReleaseAuditSentinels`.
- integration: `TestIntegrationTemporalConflictSurfaceArea` extended to
  assert `ReleaseStrict` now refuses the malformed frontier — the seam is
  closed for callers that opt in.

## 19:55 UTC — Backlog sweep PR C: Capability/Evaluation typed conflicts

The `Capability` and `Evaluation` ConflictKinds were defined in
`composition.go` but emitted by no audit — dead enum values. This wires
them into the per-side merge audit.

### Approach: classify attr disagreements by vertex type

Principled rather than reserved-key-based: in `perSideAudit`, when both
`Edited` frontiers carry the same vertex ID with the *same* type and
disagreeing attrs, the conflict kind is chosen from the vertex's ontology
type:

- `ontology.Evaluation` → `Evaluation` conflict + `EvaluationPayload`
- `ontology.Capability` → `Capability` conflict + `CapabilityPayload`
- anything else → `Textual` (unchanged)

So branches disagreeing on an evaluation result surface as an Evaluation
conflict; disagreeing on a capability definition, a Capability conflict.
Type-divergent vertices already produce a Schema conflict, so their attr
diffs stay Textual (the classifier only fires when `lt == rt`).

New `attrConflict` helper in `engine.go`; new `CapabilityPayload` /
`EvaluationPayload` in `resolvers.go` (same shape as `TextualPayload`,
distinct types so resolvers can route). No stock auto-resolver ships for
these kinds — like `Schema`/`Structural`, resolution is domain-specific;
callers can write resolvers against the typed payloads.

### Tests

`TestStrictPerSideEvaluationConflict` (asserts kind + typed payload, and
that it is *not* also Textual), `TestStrictPerSideCapabilityConflict`,
`TestStrictPerSideAttrStaysTextualForArtifact`.

### Docs

kinds. No new UC (this completes conflict-kind coverage within UC-S03).
