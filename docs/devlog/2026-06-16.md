## 19:55 UTC — Backlog sweep PR C: Capability/Evaluation typed conflicts

The `Capability` and `Evaluation` ConflictKinds were defined in
`composition.go` but emitted by no audit — dead enum values. This wires
them into the per-side merge audit.

### Approach: classify attr disagreements by vertex type

Principled rather than reserved-key-based: in `perSideAudit`, when both
`Edited` frontiers carry the same vertex ID with the *same* type and
disagreeing attrs, the conflict kind is chosen from the vertex's ontology
type:

- `ontology.Evaluation` → `Evaluation` conflict + `EvaluationPayload`
- `ontology.Capability` → `Capability` conflict + `CapabilityPayload`
- anything else → `Textual` (unchanged)

So branches disagreeing on an evaluation result surface as an Evaluation
conflict; disagreeing on a capability definition, a Capability conflict.
Type-divergent vertices already produce a Schema conflict, so their attr
diffs stay Textual (the classifier only fires when `lt == rt`).

New `attrConflict` helper in `engine.go`; new `CapabilityPayload` /
`EvaluationPayload` in `resolvers.go` (same shape as `TextualPayload`,
distinct types so resolvers can route). No stock auto-resolver ships for
these kinds — like `Schema`/`Structural`, resolution is domain-specific;
callers can write resolvers against the typed payloads.

### Tests

that it is *not* also Textual), `TestStrictPerSideCapabilityConflict`,
`TestStrictPerSideAttrStaysTextualForArtifact`.

## 20:30 UTC — Backlog sweep PR D: edge-level three-way merge

UC-U18 originally reconciled only vertices (the frontier is a vertex-ID
set). This adds edge reconciliation for `projection.Edited` frontiers,
keyed by `EdgeID` over their `EdgeEdits` maps.

### Implementation

New `(*DefaultEngine).reconcileEdges(ancestor, left, right)` in
`threeway.go` applies the same ancestor-relative decision table as the
vertex pass:

- only one side changed → take that side; agreed change → take it;
- added one side → include; added both equal → include;
- deleted one side, other unchanged → honor deletion; deleted both → omit;
- both changed differently / added-differently / modify-delete →
  `Structural` conflict (with `StructuralPayload` on type divergence).

Edge content equality compares type, endpoints, and attrs (via the
engine's Attrs equivalence). Only `Edited` frontiers carry edges, so
plain frontiers contribute nothing (edge reconciliation is a no-op) —
consistent with the presence-only degradation of the vertex pass. The
chosen edges are written into the merged `EditedFrontier.Edges` map;
edge conflicts join the same conflict slice, so any edge conflict makes
the whole merge return conflicts (merged-xor-conflicted preserved).

### Tests

`...EdgeAddition`, `...EdgeDeletionHonored`, `...EdgeModifyDelete`.

## 21:50 UTC — Backlog sweep PR F: durable namespace FileStore (UC-S22)

Final backlog item: a real persistent backend. Placed behind
`namespace.Store` — the sole mutable component of the architecture — so
persisting it is the meaningful durability boundary. The graph is
content-addressed and reconstructable, so it does not need its own store
to make a repository durable.

### namespace.FileStore (internal/namespace/file.go)

A concurrency-safe, file-backed `Store`:

- JSON file mapping each name (ref/alias/projection) to a hex-encoded
  vertex ID.
- Every method holds a mutex — unlike `memStore`, which the architecture
  doc flags as unsafe for concurrent writers. This closes that caveat.
- Each `Bind*` flushes the whole state with an atomic write-then-rename
  (`.tmp` + `os.Rename`), so a crash mid-write cannot corrupt the store.
- `NewFileStore(path)` loads an existing file or starts empty; a corrupt
  file surfaces an error rather than silently resetting.

### Tests (file_test.go)

Interface assertion; bind/resolve for all three name kinds;
durability-across-reopen (a second FileStore on the same path sees the
first's writes — simulated restart); rebind-overwrite; corrupt-file
error; unbound; and a 16-goroutine concurrent-writer test that passes
under `go test -race`.

### Scope note

FileStore is a library capability (a server/daemon embedding
`repo.Service` would use it). The `cmd/got` CLI keeps its own JSON
snapshot model — legitimate for a stateless command tool, and its refs
are already durable via that snapshot — so it was not rewired onto
FileStore to avoid a dual source of truth. Graph-snapshot persistence
remains a possible future step; it is not required for a durable
repository because the graph is reconstructable.

## 23:05 UTC — /loop plan PR H: repo durable state (UC-U20)

Second /loop step: facade-level persist/reload, composing the graph
codec (UC-S23) and the durable namespace FileStore (UC-S22). Gives
FileStore its first in-repo consumer.

### internal/repo/persist.go

A repository directory holds two files with two persistence rhythms:

- `graph.json` — the immutable graph value, written explicitly by
  `SaveState` (atomic write-then-rename). Changes only when an operation
  returns a new State, so it needs an explicit save.
- `namespace.json` — the mutable namespace, backed by `namespace.FileStore`
  which flushes every bind on its own. Durable the moment a bind happens.

`LoadState(dir, schema)` reads the graph (empty if absent; validated on
decode) and opens a FileStore for the namespace, returning a `State`.
`SaveState(dir, state)` persists the graph. This mirrors the architecture
split: append-only graph vs. single mutable namespace shell.

Design note: SaveState persists the graph, not the namespace — a State
from LoadState is FileStore-backed, so its bindings are already durable.
A caller who builds a State with a non-FileStore namespace is responsible
for that namespace's durability (documented on SaveState).

## 01:40 UTC (next day) — first-class branches (UC-U21)

Direct user direction, after a discussion about where git falls short:
git branches are bare mutable pointers with no identity, metadata, or
history, and git loses structural information. The ontology reserved a
`BranchSelector` vertex type for first-class branches but nothing used
it. This builds it.

### What a branch is now

A branch is a `BranchSelector` vertex in the append-only graph:
content-addressed identity (from the name), metadata attributes
(`branch.name`, optional `branch.parent`, description, …), and a
`forks_from` edge to its parent branch. It therefore persists, is
queryable, and has traceable fork ancestry. The one mutable, advancing
part — the tip — stays a namespace binding (`name -> tip vertex`), so
mutability concentrates there rather than vanishing (the honest tradeoff
named in the design discussion).

### Changes

- `internal/ontology/schema.go`: admissible triple
  `{BranchSelector, ForksFrom, BranchSelector}` (additive).
- `internal/repo/branch.go`: `Branch` struct, `BranchVID`,
  `CreateBranch` (ingest vertex + optional forks_from edge + optional tip
  bind), `Branches` (query BranchSelector vertices), `BranchLineage`
  (walk forks_from up to root). Sentinels `ErrBranchExists`,
  `ErrUnknownBranch`. Concrete methods on `*DefaultService` — no Service
  interface change.
- `cmd/got`: `branch <name> [--from --tip --desc]`, `branches`,
  `branch-log <name>`. `nameOf` falls back to `branch.name` so branch
  vertices display in `list vertices`.

### The git-can't demo

    got branch main --desc trunk
    got branch release-2 --from main
    got branch hotfix --from release-2
    got branch-log hotfix   ->  hotfix <- release-2 <- main

Git cannot answer "what did this branch fork from, transitively" — a
branch is a pointer with no parent and no metadata. Here it's a real
object with ancestry that persists in graph.json across restarts.

## 02:10 UTC (next day) — /loop round 3, PR M: operation-first commit history (UC-S26)

User set a 5m loop (cron e4320915) to build operation-first history and
beyond — directly attacking the "git loses information" critique.
Plan: M (history DAG), N (repo+CLI: commit/log, branch tips advance),
O (structural diff), then audit + stop.

### internal/history

An operation-first commit DAG:

- `Commit`: ID, Parents, Message, Actor, Consumed/Produced (the operation
  delta — the non-lossy "how"), and Snapshot (the resulting graph state,
  UC-S23).
- `NewCommit` computes a content-addressed `CommitID` from parents +
  message + actor + resulting-state element ids. The delta is annotation,
  NOT identity — same state from same parents = same commit (mirrors git:
  id = hash(tree, parents, author, message)).
- `Log`: append-only DAG; `Add` (validates parents present), `Get`,
  `Commits` (insertion order), `Ancestors` (BFS from a commit through
  parents, deduped across merge parents — like `git log`).
- `Marshal`/`Unmarshal`: JSON persistence with hex ids.

Why it beats git here: git records a snapshot and reconstructs intent
heuristically; this records the operation alongside the state, so history
is non-lossy. Storage note: each commit carries a full snapshot today;
dedup (git's tree/blob sharing) is a future optimization.

### Tests

Content-addressing (same inputs → same id; changed message/state → new
id; delta not in id), merge commit + BFS ancestry, unknown-parent,

## 03:00 UTC (next day) — /loop round 3, PR O: structural diff (UC-S27)

Last feature of the operation-first-history plan: a structure-aware diff.

### internal/graph/diff.go

`graph.Diff(old, new Snapshot) Delta` matches elements by ID and
classifies each: added (new only), removed (old only), changed (both,
differing content). `Delta` carries added/removed/changed vertices and
edges; `Empty()` reports no change. Changed is meaningful here because a
VertexID is sha256(name), so the same id can carry different Type/Attrs
across versions — a semantic change, not a remove+add. Reversible:
Diff(a,b) swaps added/removed vs Diff(b,a).

This is the opposite of git's line diff: reordering/moving doesn't
register as change, and independent edits don't collide.

### CLI

`diff <branch>` (last commit vs its parent) and `diff <a> <b>` (two
branch heads), rendering + / - / ~ lines with names recovered from the
snapshot's got.name attrs.

## 04:15 UTC (next day) — /loop round 4, PR Q: semantic merge + merge-base (UC-U24)

The headline VCS operation: merge a branch into the current one with a
semantic (not textual) three-way merge.

### repo.MergeStates (internal/repo/merge.go)

Three-way merges two committed snapshots (left/right) against their
common-ancestor snapshot (base) via the composition engine (UC-U18):
builds Edited frontiers carrying every vertex+edge of each snapshot,
runs MergeThreeWay, and rebuilds the merged graph from the result
frontier (dropping edges whose endpoints didn't survive). Clean →
merged graph; divergent same-target edits → typed conflicts.

### history.MergeBase

Nearest common ancestor of two commits (BFS ancestors of a, first hit
walking b) — the three-way merge base.

is an ancestor; "already up to date" if other is an ancestor; else

## 11:30 UTC — PR: chunk-level file merge (UC-U36)

Follow-up to the "can the graph enhance chunk merges?" question. Answer: yes,
by reusing the vertex merge engine one level finer — no bespoke diff3.

- `cmd/got/chunk.go`: `chunker` interface + `blockChunker` (tier-2,
  parser-free). Splits a file into top-level brace blocks and standalone lines,
  each keyed by its signature (first non-blank line, occurrence-disambiguated)
  so a body edit keeps the chunk's identity. Split→Join is the identity.
- `cmd/got/chunkmerge.go`: `reconcileFilesByChunk` pre-pass in `merge`. For each
  file changed on both sides vs base, decomposes base/ours/theirs into chunk
  vertices and runs them through `repo.MergeStates` (the composition pushout) at
  chunk granularity. Clean → rewrites both sides to the reassembled merged file,
  dissolving the file-level conflict; conflicted chunks are left for the
  file-level merge (still resolvable via --ours/--theirs).

Honest boundary, established empirically before writing the claim: git already
merges *disjoint body edits* cleanly (tested — S1/S3 clean). The real win is
**same-location structural additions** — two branches each add a new function at
EOF: git CONFLICTs on the shared insertion point, Got keeps both (verified
head-to-head). Correctness guard: same-chunk divergent edits still conflict, not
silently merged.

Limits documented in the UC: parser-free brace counting (ignores braces in
strings/comments); insertion-ordering when both sides insert at one point is a
deterministic heuristic. Natural next step: a `go/ast` or tree-sitter chunker
behind the same `chunker` interface for semantic granularity.
