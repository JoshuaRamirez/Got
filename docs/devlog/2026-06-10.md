### Bug fixed: realization.Materialize did not enforce fidelity axiom

UC-S14 declares:

> Axiom: materialize(G, P, T) = ok(B) ∧ path ∈ Paths(B)
>   ⇒ Provenance(path) ⊆ vertexIDs(project(G, P))

`realization.DefaultEngine.Materialize` was delegating to the
registered materializer and returning whatever it produced — including
provenance that escaped the view. The stock `manifestMaterialize` and
`jsonManifestMaterialize` happen to satisfy the axiom, but any custom
materializer (registered via `Register`) could violate it silently.

Fix: after the materializer returns, build the view's vertex-ID set
and walk every bundle path's provenance; on any escape, return an
error wrapping `ErrTargetUnsupported` naming the offending path.

Test: `TestMaterializeRejectsProvenanceEscape` registers a deliberately
leaky materializer and verifies the error.

### Real seam documented (not fixed): Strict audit does not run on Release

`TestIntegrationTemporalConflictSurfaceArea` documents that
`repo.Service.Release` calls `governance.GateRelease` directly and
does NOT run the Strict-mode composition audit. A vertex with a
malformed `TimeTriple` surfaces as a Temporal conflict through Merge
but not through Release.

Whether that is a bug or a feature depends on UC intent. UC-U07
(Promote release) only mentions governance gating; it does not say
the temporal/structural audits must run. So today's behavior matches
the UC. The test documents the seam explicitly so a future change
that wants Strict-on-Release knows where to look.

### What I didn't do (and why)

- **Cron scheduling.** The `/loop` invocation asked for a 1m recurring
  schedule. The `CronCreate` tool isn't available in this sandbox;
  `ToolSearch` for cron returns no match. Ran the prompt once instead.
  Re-invoke `/loop` if the cron tool gets wired up later.
- **`Revise` exposing the `ChangeCapsule`.** `repo.Service.Revise`
  discards the capsule returned by `revision.Engine.Apply`. The
  Service interface returns `(State, error)`; the capsule has nowhere
  to go. Changing the interface to return the capsule would force
  every caller and test to update. Left as a known limitation; UC-U02
  step 6 says "System emits a ChangeCapsule" but does not require it
  to be returned to the caller.
- **Performance regression tracking.** Benchmarks exist but aren't run
  in CI. Out of scope for a test-coverage pass.
- **Property-based / generative testing.** Provenance axioms have
  fixture-driven tests already; widening to property tests would be
  net new scope.

## 02:40 UTC — Resolver library completion + ReviseWithCapsule

Continued from the integration/BDD PR (#20). Three changes:

### 1. Pluggable Attrs equivalence

`composition.DefaultEngine` gains an `AttrsEqualFunc` field and a
`SetAttrsEqual` method. The default (`DefaultAttrsEqual`) preserves
the existing bitwise behavior. Callers that need semantic equivalence
(canonical JSON, set comparison on slices, etc.) supply their own
predicate.

Closes one of the documented decisions from the 2026-05-17 juncture
(equivalence predicates). The implementation is additive and
behavior-preserving by default.

Test added: `TestSetAttrsEqualTolerant` proves a tolerant predicate
suppresses the Textual conflict; `TestSetAttrsEqualNilResets` proves
nil restores the default behavior.

### 2. Four more stock resolvers

| Resolver | Kind | Behavior |
|---|---|---|
| `PreferRightAttr(key)` | `Textual` | Mirror of `PreferLeftAttr`. |
| `PreferLowerTrust()` | `Trust` | Mirror of `PreferHigherTrust`. |
| `PreferEarlierTime()` | `Temporal` | Keep the earlier `ValidFrom`. |
| `PreferLaterTime()` | `Temporal` | Keep the later `ValidFrom`. |
| `RejectSchemaConflict()` | `Schema` | Refuses Schema conflicts; surfaces `ErrConflictUnresolvable`. |

`Trust` and `Temporal` resolvers share a small internal type each
(`preferTrustResolver`, `preferTimeResolver`) parameterized by a
direction flag.

`RejectSchemaConflict` is meaningfully different: it returns an error
rather than mutating frontiers, on the principle that Type changes
are semantically dangerous and should fail loudly rather than persist
as unresolved noise.

Tests added for each resolver. Library completeness for the per-side
conflict kinds is now: Schema (reject), Textual (left/right), Trust
(higher/lower), Temporal (earlier/later), Structural (no
auto-resolver — callers must supply a custom one).

### 3. `repo.Service.ReviseWithCapsule`

The Service interface had a quiet gap: `Revise` discarded the
`ChangeCapsule` that `revision.Engine.Apply` returns. UC-U02 step 6
declares "System emits a `ChangeCapsule` recording consumed and
produced vertex IDs," and UC-U14 (Replay) requires the capsule as
input. The facade made the capsule inaccessible.

Added `ReviseWithCapsule(ctx, state, rule, match) → (State, ChangeCapsule, error)`.
`Revise` now delegates to it and discards the capsule, preserving the
backward-compatible signature.

Tests added: `TestReviseWithCapsule` confirms the capsule reports
correct `Produced` IDs and an empty `Consumed` for a pure-addition
rewrite. `TestReviseDelegatesToReviseWithCapsule` confirms the
backward-compatible wrapper still works.

### What I did NOT do

- **Schema auto-resolver beyond Reject.** Type changes between
  branches are semantically dangerous (Artifact↔Revision means
  different things). The only safe auto-resolution would be
  caller-supplied; offering one in the library would encourage
  unsafe practice.
- **Symmetric Strict bridge for `revision.Apply`.** Still on the
  menu; not started.
- **Three-way `MergeThreeWay`.** Still on the menu.
- **`Revise` returning the capsule directly (not via a parallel
  method).** Would change the Service interface signature and break
  every existing caller. The parallel-method approach preserves
  backward compatibility.
