### Bug fixed: realization.Materialize did not enforce fidelity axiom

UC-S14 declares:

> Axiom: materialize(G, P, T) = ok(B) ∧ path ∈ Paths(B)
>   ⇒ Provenance(path) ⊆ vertexIDs(project(G, P))

`realization.DefaultEngine.Materialize` was delegating to the
registered materializer and returning whatever it produced — including
provenance that escaped the view. The stock `manifestMaterialize` and
`jsonManifestMaterialize` happen to satisfy the axiom, but any custom
materializer (registered via `Register`) could violate it silently.

Fix: after the materializer returns, build the view's vertex-ID set
and walk every bundle path's provenance; on any escape, return an
error wrapping `ErrTargetUnsupported` naming the offending path.

Test: `TestMaterializeRejectsProvenanceEscape` registers a deliberately
leaky materializer and verifies the error.

### Real seam documented (not fixed): Strict audit does not run on Release

`TestIntegrationTemporalConflictSurfaceArea` documents that
`repo.Service.Release` calls `governance.GateRelease` directly and
does NOT run the Strict-mode composition audit. A vertex with a
malformed `TimeTriple` surfaces as a Temporal conflict through Merge
but not through Release.

Whether that is a bug or a feature depends on UC intent. UC-U07
(Promote release) only mentions governance gating; it does not say
the temporal/structural audits must run. So today's behavior matches
the UC. The test documents the seam explicitly so a future change
that wants Strict-on-Release knows where to look.

## 02:40 UTC — Resolver library completion + ReviseWithCapsule

Continued from the integration/BDD PR (#20). Three changes:

### 1. Pluggable Attrs equivalence

`composition.DefaultEngine` gains an `AttrsEqualFunc` field and a
`SetAttrsEqual` method. The default (`DefaultAttrsEqual`) preserves
the existing bitwise behavior. Callers that need semantic equivalence
(canonical JSON, set comparison on slices, etc.) supply their own
predicate.

Closes one of the documented decisions from the 2026-05-17 juncture
(equivalence predicates). The implementation is additive and
behavior-preserving by default.

Test added: `TestSetAttrsEqualTolerant` proves a tolerant predicate
suppresses the Textual conflict; `TestSetAttrsEqualNilResets` proves
nil restores the default behavior.

### 2. Four more stock resolvers

| Resolver | Kind | Behavior |
|---|---|---|
| `PreferRightAttr(key)` | `Textual` | Mirror of `PreferLeftAttr`. |
| `PreferLowerTrust()` | `Trust` | Mirror of `PreferHigherTrust`. |
| `PreferEarlierTime()` | `Temporal` | Keep the earlier `ValidFrom`. |
| `PreferLaterTime()` | `Temporal` | Keep the later `ValidFrom`. |
| `RejectSchemaConflict()` | `Schema` | Refuses Schema conflicts; surfaces `ErrConflictUnresolvable`. |

`Trust` and `Temporal` resolvers share a small internal type each
(`preferTrustResolver`, `preferTimeResolver`) parameterized by a
direction flag.

`RejectSchemaConflict` is meaningfully different: it returns an error
rather than mutating frontiers, on the principle that Type changes
are semantically dangerous and should fail loudly rather than persist
as unresolved noise.

Tests added for each resolver. Library completeness for the per-side
conflict kinds is now: Schema (reject), Textual (left/right), Trust
(higher/lower), Temporal (earlier/later), Structural (no
auto-resolver — callers must supply a custom one).

### 3. `repo.Service.ReviseWithCapsule`

The Service interface had a quiet gap: `Revise` discarded the
`ChangeCapsule` that `revision.Engine.Apply` returns. UC-U02 step 6
declares "System emits a `ChangeCapsule` recording consumed and
produced vertex IDs," and UC-U14 (Replay) requires the capsule as
input. The facade made the capsule inaccessible.

Added `ReviseWithCapsule(ctx, state, rule, match) → (State, ChangeCapsule, error)`.
`Revise` now delegates to it and discards the capsule, preserving the
backward-compatible signature.

Tests added: `TestReviseWithCapsule` confirms the capsule reports
correct `Produced` IDs and an empty `Consumed` for a pure-addition
rewrite. `TestReviseDelegatesToReviseWithCapsule` confirms the
backward-compatible wrapper still works.

### What I did NOT do

- **Schema auto-resolver beyond Reject.** Type changes between
  branches are semantically dangerous (Artifact↔Revision means
  different things). The only safe auto-resolution would be
  caller-supplied; offering one in the library would encourage
  unsafe practice.
- **Symmetric Strict bridge for `revision.Apply`.** Still on the
  menu; not started.
- **Three-way `MergeThreeWay`.** Still on the menu.
- **`Revise` returning the capsule directly (not via a parallel
  method).** Would change the Service interface signature and break
  every existing caller. The parallel-method approach preserves
  backward compatibility.

## 02:50 UTC — revision Strict bridge step 1: dangling-edge detection

flagged in the 2026-05-17 juncture as the parallel to the composition
bridge. This is step 1 of the same shape: a Strictness flag and the
in-graph audit that the available API can express without an

### What changed

  behavior) and `Strict` constants.
- `revision.NewEngine()` stays Lenient. `revision.NewEngineStrict()`
  added for Strict.

3. ⏳ Hyperedge sub-pattern matching — not in scope (existing impl

## 03:20 UTC — Property-based tests for closure and merge axioms

First step of the four-part workflow (property tests → revision Strict
bridge step 2 → three-way merge → cmd CLI). This is hardening of
existing Verified UCs, not a new feature, so no new UC — the ledger's
"Next-bite candidates" already lists generative testing under Hardening.

### What was added

- `internal/provenance/provenance_props_test.go` — generates 300 random
  causal graphs (Artifact pool + random `DerivedFrom` edges, the one
  admissible homogeneous causal triple) and asserts the closure-operator
  axioms across the population:
  - extensive (`S ⊆ Close(S)`), idempotent, monotone;
  - `Cone(v) = Close({v})`;
  - `Close(S) = ⋃ Cone(s)` (closure is additive over seeds);
  - `Causes(a,b) ⟺ b ∈ Close({a})` and `Causes` is symmetric
    (undirected traversal);
  - `Causes` reflexive;
  - every `TraceSet` result is a simple path inside `Close({from})`, and
    `TraceSet` is non-empty iff the endpoints are causally connected.
- `internal/composition/composition_props_test.go` — generates 300
  random frontier pairs over an Artifact pool and asserts the
  `MergeResult` axioms:
  - Lenient merge frontier = set-union of inputs;
  - frontier commutativity (as a set) and idempotence (`Merge(f,f)=f`);
  - witness ID determinism for identical inputs;
  - the merged-xor-conflicted invariant in both directions (empty policy
    set → merged, no conflicts; `Unsat` policy → one `Policy` conflict,
    no frontier).

### Reproducibility

Each iteration seeds `math/rand` with its loop index and the failure
messages print the seed, so any property failure is replayable by
fixing the seed.

### Note documented, not asserted

`composition.deterministicWitnessID` hashes union IDs in input order,
not sorted order, so `Merge(L,R)` and `Merge(R,L)` can yield different
witness IDs even though the frontier *set* is identical. The
commutativity property therefore asserts set-equality of the frontier
and deliberately does not assert witness equality. Witness determinism
is covered separately for identical-order inputs.

## 03:55 UTC — revision Strict bridge step 2: content-addressing / identity-collision

Second step of the workflow. Closes the produce-side half of
`revision.Apply`'s spec/impl divergence — simplification #2 from the
2026-05-17 juncture ("caller-declared R-side IDs vs. content-addressed").

### The hazard

`graph.WithVertex`/`WithEdge` replace in place on a duplicate ID
(`internal/graph/mem.go:130`). So a rewrite whose Right-hand side
declares a produced vertex `{ID: X, Type: Model}` silently clobbers an
existing host `{ID: X, Type: Artifact}`. Content-addressed identity
requires equal IDs to imply equal content; a silent overwrite violates
that law (and the monotone-extension axiom). Lenient `Apply` trusted
the caller's declared IDs and let the overwrite happen.

### What "content-addressed" means here

`graph.Vertex` has no `CanonicalBytes`; in practice IDs are
`sha256(name)`, not a hash of the full vertex content. So step 2 cannot
mean "re-derive each ID from a canonical Vertex encoding" without
rejecting every existing fixture. The faithful, testable
interpretation that fits the codebase: **a produced (R-side) ID may not
collide with structurally different host content.** That is the precise
content-addressing law the Lenient overwrite breaks.

### Implementation (`internal/revision/engine.go`)

- New sentinel `ErrIdentityCollision`.
- `contentCollisions(post, right, ctxVerts, ctxEdges)` compares each
  produced non-context vertex/edge against the **post-deletion** graph;
  a same-ID-different-content match is a collision. Context (K) elements
  are excluded (preserved interface). Idempotent re-statements (same ID,
  identical content) are allowed. IDs deleted earlier in the same
  rewrite are absent from the post-deletion graph, so delete-then-add of
  the same ID is allowed (legitimate replace, not overwrite).
- `vertexContentEqual` / `edgeContentEqual` / `attrMapEqual` (nil and
  empty AttrMap treated equal).
- Strict `Apply` runs the audit after `rebuildWithout`, before the
  R-side insertions; on any collision it refuses with
  `ErrIdentityCollision` and leaves the input graph unchanged. Lenient
  is unchanged.
- Hyperedges are not audited: the current `Apply` never inserts R-side
  hyperedges, so there is no overwrite to guard against (noted in code).

### Tests added (`internal/revision/revision_test.go`)

- `TestApplyLenientOverwritesOnIdentityCollision` — documents the
  historical silent overwrite.
- `TestApplyStrictRefusesVertexIdentityCollision` — Strict returns
  `ErrIdentityCollision`; input graph unchanged.
- `TestApplyStrictRefusesEdgeIdentityCollision` — same for an edge ID
  reused with different endpoints.
- `TestApplyStrictAllowsIdempotentRestatement` — same ID + identical
  content is allowed.
- `TestApplyStrictAllowsReplaceAfterDelete` — delete-then-add of the
  same ID with new content is allowed.
- `TestErrIdentityCollisionSentinel`.
