# 2026-05-17

## 23:35 UTC — Decision juncture recorded: composition.Merge spec/impl divergence and the Lenient/Strict path forward

After a long discussion of where `composition.Merge` and `revision.Apply`
sit relative to their categorical specs, recording the analysis and the
recommended path so the choice is reversible and the reasoning is not
lost.

### Where things stand today

`composition.Merge` (`internal/composition/engine.go:30-63`) implements a
**set-union + governance-gate** semantics:

1. Union of left and right `Frontier.VertexIDs()`.
2. `governance.Engine.Check` against the union.
3. If `Sat` → witness (SHA-256 of union) + certificate via
   `verification.Engine.Certify`.
4. If `Unsat` → return `MergeResult` with a single `Policy`-kind
   `Conflict`.

This is the simplest interpretation of the UC-S03 contract that passes
all the named failure-paths. It is **correct for non-overlapping
merges** (each branch adds disjoint vertices) and **silently accepts**
overlapping merges where left and right disagree on a vertex's contents
(type, attrs, time, trust) or on edge types over a shared `(from, to)`
pair.

Six of eight `ConflictKind` constants (`Textual`, `Structural`, `Schema`,
`Trust`, `Capability`, `Evaluation`, `Temporal`) are declared in the
package but never produced.

`revision.Apply` (`internal/revision/engine.go`) follows the same shape:
literal "delete L\K, keep K, add R\K" semantics rather than constructing
a pushout complement. Dangling edges are silently dropped; R-side IDs
are caller-declared rather than derived from canonical bytes.

### The analytical frame vs. the medium of information

Two things were tangled in the spec wording. Separating them:

**Analysis (categorical vocabulary)**:
- A frontier is a subobject (mono) of the graph.
- The intersection of left and right is the pullback.
- The merge is the pushout of left and right along their pullback.
- "Guarded pushout in `Repo_Π`" means the pushout must land inside the
  policy subcategory.
- The conflict monad `T(X) = X + Conf_Π` packages the XOR of success
  and conflict into one return type.

**Medium (where information actually lives)**:

Every vertex carries information in seven distinct media, and the
conflict typing maps one-to-one to these media:

| Medium | Carrier | Conflict when | `ConflictKind` |
|---|---|---|---|
| Identity | `Vertex.ID` | Set union resolves; no conflict in isolation | — (foundation) |
| Type | `Vertex.Type` | Same ID, different `VertexType` | `Schema` |
| Attributes | `Vertex.Attrs` | Same key, different values | `Textual` |
| Time | `Vertex.Time` | `ValidFrom`/`ValidTo` or `CausalTime` disagree | `Temporal` |
| Trust | `Vertex.Trust` | `Score`/`Class` divergence violates policy | `Trust` |
| Edges | `graph.Edge` | Same `(from, to)` with incompatible types | `Structural` |
| Hyperedges | `graph.Hyperedge` | Overlapping N-ary relations | `Structural` |

Plus two higher-layer media:

| Medium | Carrier | `ConflictKind` |
|---|---|---|
| Evaluation | attached `verification.Evaluation` | `Evaluation` |
| Capability | computed `capability.Witness` | `Capability` |

And the orthogonal guard medium:

| Medium | Carrier | `ConflictKind` |
|---|---|---|
| Policy | `governance.Policy` set | `Policy` |

The current impl operates on only **two** media: Identity (collapsed to
set union, no content check) and Policy (the governance gate). The
other seven are blind spots.

### Trade-off summary

| Axis | Current (union + gate) | Full per-medium |
|---|---|---|
| Code size | ~150 LOC | ~400–600 LOC |
| Runtime | O(n) | O(n·m), m=media checked; still linear |
| Conflict kinds reachable | 1 of 8 | 7 of 8 (plus `Capability` from a different engine) |
| False negatives | High — accepts mismatched contents | Low |
| False positives | None | Possible (semantically-equivalent attrs flagged) |
| API stability | Stable | Likely needs Frontier to carry per-side edits or Merge to take an ancestor |
| Test rework | None | ~half the merge tests need re-evaluation |
| Witness ID stability | `hash(union)` | `hash(pushout)` — breaks stored witnesses |
| Resolver power | Generic `Apply(g)` | Typed per-medium resolvers possible |
| Spec alignment | Off | On |

### Key decision points that any per-medium impl must answer

1. **Where do per-side vertex contents come from?**
   - (A) Inspect the host graph — only catches sequential edits.
   - (B) Frontier carries `VertexEdits` map — concurrent-edit detection,
     API change.
   - (C) Three-way merge: Merge takes a common-ancestor graph — biggest
     API change, classic git model.

2. **Equivalence predicates per medium**. Strict bitwise equality
   maximises false positives; loose equality hides real conflicts. Each
   medium needs its own rule (Time intervals: strict equal vs. interval
   intersection; Attrs: bitwise vs. canonical-JSON; etc.).

3. **Witness ID stability**. Switching from `hash(union)` to
   `hash(pushout-data)` silently breaks any external code that pinned
   the current ID. Affects `release.Service`'s alias ledger if witness
   IDs are stored there long-term.

4. **Resolver expressiveness**. Generic graph-mutation resolvers
   (`Apply(g) → g'`) are enough to "delete the offender" but can't say
   "for this `Textual` conflict on key K, take left's value." Typed
   resolvers are a new API surface to design.

5. **Conflict noise**. The full impl will say "no, 14 conflicts" much
   more often than today. Mitigation is a `MergePolicy` API for
   pre-declared resolution rules — itself a non-trivial design.

### The cheapest path that retains the option

The recommended bridge (do not yet implement):

1. Add a per-medium audit hook to `Merge` that runs after the union
   step but does not change the return type. Discovered conflicts log
   to a side channel for observability without failing the merge.
2. Add an `ext.Strictness` flag to `Merge`: `Lenient` (today's
   behavior) or `Strict` (full per-medium). Default `Lenient`. New
   callers opt in.
3. Test the strict path independently with new fixtures exercising each
   conflict kind. The lenient path keeps its current five tests.
4. Flip the default when production callers are ready, with a
   migration window.

This preserves backward compatibility, makes the choice reversible, and
lets the categorical implementation be approached incrementally.

### Why we are not implementing now

- The current impl satisfies every existing UC (ledger reads 37/37
  Verified).
- No active caller asks for the missing detection. The blind spots are
  real but unreached.
- A categorical re-impl changes observable behavior in ways that
  depend on tradeoffs the project has not yet selected (strict-on-
  textual? three-way vs two-way? witness ID stability?).
- The cheap bridge is itself non-trivial work (Strictness flag,
  per-medium predicates, conflict packaging, test fixtures) and should
  be motivated by a concrete need.

This entry is the record. The next time someone asks "why isn't
composition doing the typed conflicts" the answer is: read this and
decide which of the five decision points has an answer now.

Same logic applies to `revision.Apply` for the pushout-complement /
dangling-edge detection; the analysis is structurally parallel.
