# 2026-05-21

## 15:09 UTC — composition Strict step 2: per-side data carrier via projection.Edited

Executed step 2 of the four-step bridge from
`docs/devlog/2026-05-17.md`. Picked the Frontier-with-edits shape over
three-way-with-ancestor on three grounds: smaller API change (no new
Engine method), no need to invent a parent-pointer state model the
codebase doesn't have, and three-way can still be added later on top
of this as a separate `MergeThreeWay` method without conflicting.

### What changed

**`internal/projection/projection.go`**:

- New `Edited` interface — optional extension of `Frontier` that adds
  `VertexEdits() map[VertexID]Vertex` and `EdgeEdits() map[EdgeID]Edge`.
- New concrete `EditedFrontier` struct (with `NewEditedFrontier` ctor)
  that satisfies both `Frontier` and `Edited`.
- Existing `IDsSelector`-derived frontiers continue to work; they do
  not satisfy `Edited` and so are treated as "no per-side data
  available" by the audit.

**`internal/composition/engine.go`**:

- `perSideAudit(left, right)` — runs only when BOTH sides satisfy
  `projection.Edited`. Otherwise returns nil and the strict path falls
  back to in-graph audits only.
- For each shared vertex ID in left and right vertex edits, compares:
  - `Type` → emits `Schema` on disagreement.
  - `Trust` → emits `Trust` on bitwise disagreement.
  - `Time` → emits `Temporal` on bitwise disagreement.
  - `Attrs` per shared key → emits `Textual` on disagreement
    (`attrsEqual` does `==` with `recover` fallback on non-comparable
    types, then `fmt.Sprint` as a conservative last resort).
- For each `(from, to)` pair appearing in both sides' edge edits with
  different types → emits `Structural`.

### What this closes from the trade-off table

The earlier juncture analysis listed seven media of information the
simplified impl was blind to. Strict now covers five at the per-side
level (Schema, Textual, Trust, Temporal, Structural) when both
frontiers carry edits, plus the original two in-graph checks
(Structural via edge collisions in `g`, Temporal via malformed
`TimeTriple` on stored vertices).

Still NOT covered:

- `Capability` — emergence is a higher-layer concern; needs
  `capability.Engine` integration.
- `Evaluation` — same; needs evaluation evidence attached to the
  frontier.
- Three-way semantics — without a common ancestor, the current path
  can't distinguish "both added X" (no conflict) from "ancestor had X,
  left modified, right deleted" (real conflict). Two-way is still
  semantically weaker than three-way; the bridge doesn't close that.

### Equivalence is bitwise

The per-side audits use `==` (plus the recover/fmt fallback for
non-comparable Attrs values). This is conservative — false-positive
prone for Attrs values that are semantically equal but not bitwise
equal (e.g. JSON with different key order). Refining is one of the
open decisions from the juncture; see `docs/devlog/2026-05-17.md`
decision-point #2. Refinement can be additive: replace `attrsEqual`
with a domain-aware predicate when a real need surfaces.

Step 3 (typed resolvers) is the next natural piece. Today
`composition.Resolution` is just `Apply(g) → (g, error)` — generic
graph rewriting. Typed resolvers (`TextualResolver`, `TrustResolver`,
etc.) would let callers say "for any Textual conflict on attr K,
choose left's value" without writing a custom graph mutator. The
shape: introduce `KindedResolution` as a tagged Resolution that the
Resolve loop dispatches on `Conflict.Kind()`.

Step 4 (flip default) is the largest blast radius. The current
default `NewEngine → Lenient` keeps existing callers working unchanged.
Flipping to `NewEngine → Strict` would surface previously-silent
conflicts for any caller using `EditedFrontier` — by then, typed
resolvers (step 3) should be available so callers have a way to
handle the new conflicts. The flip itself is a one-line change plus
a migration window announcement.
